## Безопасность

### [HIGH] src/arachna/runner.py:88-93,104-108 — _resolve_command блокирует piped-команды с разрешёнными утилитами

**Статус**: новая находка. Проявилась после удаления интерпретаторов из `_ALLOWED_COMMANDS`.
**Суть**: `_resolve_command()` на строках 88-93 проверяет наличие shell-метасимволов в команде. Если они есть — возвращает команду целиком:

    def _resolve_command(cmd: str) -> str:
        cmd = cmd.strip()
        if _SHELL_CHARS.intersection(cmd):
            return cmd    # <-- возвращает "echo hello | cat" целиком
        parts = shlex.split(cmd)
        ...

Далее `_validate_command()` на строках 104-108 проверяет базовое имя против `_ALLOWED_COMMANDS`:

    base = _resolve_command(cmd)
    if base and not _SHELL_CHARS.intersection(cmd) and base not in _ALLOWED_COMMANDS:
        return False, f"command not in allowlist: '{base}'"

Условие `not _SHELL_CHARS.intersection(cmd)` защищает piped-команды от проверки по allowlist'у. Это правильное намерение: piped-команду нельзя проверить по базовому имени. Но `_BLOCKED_PATTERNS` всё равно проверяется (строка 101-105), а он содержит только явно опасные паттерны. Piped-команда из разрешённых утилит пройдёт валидацию.

Однако на строке 118 `run_command` проверяет `needs_shell`:

    needs_shell = any(c in cmd for c in _SHELL_CHARS)

Если в команде есть `|`, `needs_shell = True`, и команда выполняется через `shell=True`. Это корректно.

**Реальная проблема**: `_resolve_command` возвращает строку целиком для piped-команд, но `_validate_command` не проверяет отдельные части пайпа. Если одна из частей содержит небезопасную команду не из `_BLOCKED_PATTERNS` и не из `_ALLOWED_COMMANDS`, она пройдёт. Пример:

    echo hello | python3 -c "print('safe')"  # python3 не в _ALLOWED_COMMANDS
    # _resolve_command: "echo hello | python3 -c ..." (целиком)
    # _SHELL_CHARS.intersection: True → проверка allowlist пропущена
    # _BLOCKED_PATTERNS: "python" не в списке → пропущено
    # Результат: команда выполнена, хотя python3 не в allowlist

Но `python3` не в `_ALLOWED_COMMANDS` и не в `_BLOCKED_PATTERNS`. Раньше `python` был в `_ALLOWED_COMMANDS` и проходил. Теперь `python3` — это не `python`, и он не в `_ALLOWED_COMMANDS`. Однако из-за обхода проверки allowlist для piped-команд, `python3` будет выполнен.

**Это не RCE**, потому что `python3` всё ещё не интерпретатор `python` из удалённых. Но это обход allowlist'а: команда, которой нет в разрешённом списке, выполняется через пайп.

**Влияние сейчас**: злоумышленник может выполнить `python3` через piped-команду в `.arachna.json`. Но `python3` — это системный интерпретатор, который и так доступен. Риск ниже, чем когда `python` был в allowlist'е, потому что теперь злоумышленнику нужно чтобы `python3` был установлен и находился в PATH. На большинстве систем — да.

**Риск при росте**: любой новый интерпретатор или опасная утилита, не попавшая в `_BLOCKED_PATTERNS`, может быть выполнена через пайп с разрешённой утилитой.

**Исправление**: в `_validate_command` для команд с shell-метасимволами проверять каждую часть пайпа отдельно. Либо разбивать по `|` и проверять базовое имя каждой части против `_ALLOWED_COMMANDS`. Либо запретить shell-метасимволы в принципе (убрать `shell=True`), но это сломает легитимные pre_commands.

### [HIGH] src/arachna/splitter.py:29-35 — _handle_single возвращает was_truncated, но split() его теряет

**Статус**: новая находка.
**Суть**: `_handle_single()` (строки 83-99) возвращает `tuple[list[str], bool]` — части и флаг `was_truncated`. Вызов в `split()` на строках 29-35:

    elif mode == "single":
        parts, was_truncated = _handle_single(raw_content, max_tokens, tokenizer=tk)
        if was_truncated:
            print(
                f"  Warning: content is {tk(raw_content)} tokens, "
                f"limit {max_tokens} — truncating"
            )
        return parts

Флаг `was_truncated` используется только для печати предупреждения. Вызывающий код (`collect()` в collector.py, `dry_run()` в gatherer.py) получает только список частей — информация об усечении теряется. Вызывающий код не может отреагировать программно (например, установить exit code 1, записать в лог, пометить выходной файл).

**Влияние сейчас**: в режиме `single` при превышении лимита пользователь видит warning в stdout, но программа возвращает exit code 0. Вызывающий код (как библиотека) не знает об усечении.

**Риск при росте**: при использовании arachna как библиотеки в CI/CD или IDE-плагине усечение контекста остаётся незамеченным. AI получает неполный контекст, решения принимаются на основе усечённых данных.

**Исправление**: либо изменить сигнатуру `split()` для возврата `(parts, was_truncated)`, либо добавить отдельный метод `split_with_truncation_flag()`, либо пробрасывать информацию через warning-логгер вместо print. Минимальное исправление: логировать через `logging.warning` вместо `print`, чтобы библиотечный пользователь мог перехватить.

### [MEDIUM] src/arachna/runner.py:127-130 — shlex.split без проверки на пустой список аргументов

**Статус**: новая находка.
**Суть**: `run_command()` на строках 127-130:

    args = shlex.split(cmd)
    result = subprocess.run(args, capture_output=True, text=True, timeout=30)

Если `cmd` — пустая строка или только пробелы, `shlex.split("")` возвращает `[]`. `subprocess.run([])` выбрасывает `ValueError` (в Python 3.11+: "must have at least one argument"). Это исключение попадает в `except` на строке 132 и подавляется — `_log_command(cmd, False)` и возврат `""`. Функционально корректно (пустая команда → пустой вывод), но семантически грязно: ошибка валидации маскируется под ошибку выполнения.

Кроме того, `shlex.split` может выбросить `ValueError` при непарных кавычках. Например, `shlex.split('echo "hello')` → `ValueError: No closing quotation`. Это тоже попадёт в except и вернёт `""`. Пользователь не узнает о синтаксической ошибке в команде.

**Влияние сейчас**: низкое. Команды в `.arachna.json` обычно валидны.

**Риск при росте**: при поддержке пользовательских команд через CLI-аргументы (`--command`) синтаксические ошибки будут молча игнорироваться, запутывая пользователя.

**Исправление**: проверять `if not cmd.strip(): return ""` до `shlex.split`. Завернуть `shlex.split` в отдельный try/except с `ValueError` и логировать предупреждение о синтаксической ошибке.
