## HIGH

### H-01: Race condition в операциях с манифестом и файлами кэша
**Область:** архитектура / целостность данных
**Файлы:** `src/arachna/collector.py:88-101`, `src/arachna/cache.py:17-19`
**Суть:** `_cmd_single` вызывает `clean_manifest`, затем `collect`, затем `save_manifest` — три раздельных операции без блокировок. При параллельном запуске двух `--profile` возможна гонка: первый процесс очищает манифест, второй перезаписывает его пустым состоянием, затем первый сохраняет свой список файлов, теряя файлы второго. Аналогично `save_cache` делает `write_text` без атомарности — крах во время записи оставит повреждённый JSON.
**Влияние сейчас:** низкое (CLI-утилита, редко запускается параллельно).
**Риск при росте:** средний — при интеграции в CI/CD пайплайны параллельные запуски станут реальностью, потеря манифеста приведёт к orphan-файлам в output-директории и невозможности clean.

### H-02: Shell-инъекция через поля конфига без аудит-лога
**Область:** безопасность
**Файлы:** `src/arachna/runner.py:17-32`, `src/arachna/gatherer.py:36`, `src/arachna/collector.py:131-133`
**Суть:** `run_command` выполняет произвольные shell-команды из `pre_commands`, `command` и `post_commands`. Это задокументировано в README как осознанное решение ("Only use .arachna.json from trusted sources"), но отсутствует какой-либо журнал исполненных команд. При компрометации конфига (например, через PR в репозиторий) злоумышленник может выполнить произвольный код, и расследование инцидента будет затруднено — нет записи что именно было выполнено.
**Влияние сейчас:** пользователь выполняет свой же конфиг — риск низкий.
**Риск при росте:** высокий — если arachna начнут запускать в automated-окружениях (CI, pre-commit хуки), подмена `.arachna.json` станет вектором атаки без возможности аудита.

## MEDIUM

### M-01: Подхват .gitignore из исключённых директорий неполный
**Область:** безопасность / сбор контента
**Файлы:** `src/arachna/gitignore.py:11-14`
**Суть:** Фильтр исключает директории по жёстко закодированному списку `venv`, `node_modules`, `__pycache__`, и `.`. Но `.gitignore` в `.tox`, `.mypy_cache`, `.pytest_cache` и других инструментальных директориях будет загружен и может добавить легитимные файлы в исключения или наоборот — пропустить sensitive-файлы если в локальном `.gitignore` кривые правила.
**Влияние сейчас:** низкое — большинство проектов не имеют вложенных `.gitignore` в этих директориях.
**Риск при росте:** средний — с ростом числа поддерживаемых экосистем (JS, Rust, Go) список жёстко закодированных исключений будет отставать, что приведёт к утечке временных файлов или артефактов сборки.

### M-02: Глобальный state в модуле токенизатора
**Область:** архитектура / тестопригодность
**Файлы:** `src/arachna/gatherer.py:16-21`
**Суть:** Токенизатор хранится в глобальной переменной `_TOKENIZE` модуля `gatherer.py` и устанавливается через `set_tokenizer`. Это означает что два разных вызова `collect` с разными профилями в одном процессе будут использовать токенизатор последнего установленного профиля. Токенизатор также импортируется напрямую в `splitter.py` (строка 3) — минуя глобальный state, что создаёт несогласованность: splitter всегда использует дефолтный, а gatherer — последний установленный.
**Влияние сейчас:** на практике пользователь запускает CLI однократно, профиль один — не проявляется.
**Риск при росте:** высокий — если arachna будет использоваться как библиотека (импорт `collect` напрямую), подсчёт токенов станет некорректным при последовательных вызовах с разными профилями. Splitter будет недооценивать токены, что приведёт к превышению контекстного окна модели.

### M-03: Truncate в single-режиме теряет контент без возможности восстановления
**Область:** архитектура
**Файлы:** `src/arachna/splitter.py:94-100`
**Суть:** `_handle_single` при превышении `max_tokens` обрезает текст по `max_chars = max_tokens * 4` и добавляет `# ... truncated ...`. Это необратимая операция — нет ни флага `truncated` в возвращаемом значении, ни записи сколько контента потеряно. Вызывающий код в `collect` не узнает что контент был обрезан.
**Влияние сейчас:** пользователь видит warning в stdout, но в CI логах может потеряться.
**Риск при росте:** средний — при автоматизации сбора контекста для AI-агентов, передача обрезанного контента без явного признака приведёт к тому что модель будет работать с неполными данными и не сообщит об этом.

### M-04: Тесты используют `os.chdir` и `patch('pathlib.Path.cwd')` одновременно
**Область:** тестопригодность
**Файлы:** `tests/gatherer/test_collect_sections.py:11-19`, аналогично в `test_gather_files`, `test_dry_run`, `test_incremental`
**Суть:** Тесты делают `os.chdir(d)` и одновременно патчат `Path.cwd()` — это избыточно и создаёт риск: если патч не сработает в какой-то среде, тест упадёт с неожиданной ошибкой. Более того, `os.chdir` в тестах — глобальная операция, которая влияет на все последующие тесты если `finally` не выполнится (например при SIGTERM).
**Влияние сейчас:** тесты проходят, но зависят от порядка выполнения.
**Риск при росте:** средний — при распараллеливании тестов (`pytest-xdist`) `os.chdir` сломает изоляцию, тесты начнут падать недетерминированно.

## LOW

### L-01: Жёстко закодированные пути в init.py не расширяемы
**Область:** архитектура
**Файлы:** `src/arachna/init.py:7-21`
**Суть:** Списки `_CODE_DIRS`, `_TEST_DIRS`, `_DOCS_DIRS` зашиты в коде. Добавление поддержки нового языка (например, Elixir с `lib/` и `mix.exs`) требует изменения исходного кода. Для CLI-утилиты это приемлемо, но ограничивает сообщество в контрибуции.
**Влияние сейчас:** нулевое.
**Риск при росте:** низкий — можно вынести в конфиг или плагины при достижении ~10+ поддерживаемых языков.

### L-02: Отсутствует тест на параллельный запуск manifest-операций
**Область:** тестопригодность
**Файлы:** `tests/collector/test_collect.py` — нет тестов параллелизма
**Суть:** Проблема H-01 не покрыта тестами. Нет теста который симулирует два конкурирующих вызова `collect` и проверяет целостность манифеста.
**Влияние сейчас:** баг не проявляется в одиночных тестах.
**Риск при росте:** низкий — но станет MEDIUM как только появится CI-интеграция.

## СИЛЬНЫЕ СТОРОНЫ

1. **Чёткое разделение ответственности:** gatherer (сбор) → collector (оркестрация) → splitter (разбиение) → formatter (форматирование) — каждый модуль тестируется изолированно.
2. **Плагинная архитектура токенизатора:** `load_tokenizer` с конвенцией `module:function` — расширяемо без изменения ядра.
3. **Инкрементальный режим на mtime:** корректно обрабатывает new/changed/deleted, кэш отделён от манифеста.
4. **179 тестов / 90% покрытия:** отличная база, большинство модулей покрыты изолированными юнит-тестами.
5. **Безопасное сжатие:** только пробелы и пустые строки, индентация не трогается — правильный баланс между экономией токенов и безопасностью.

## ВЕРДИКТ

Самое опасное — **race condition в операциях с манифестом (H-01)** и **отсутствие аудит-лога shell-команд (H-02)**. 

С чего начать:
1. H-01: Добавить атомарную запись через tempfile + rename для манифеста и кэша
2. H-02: Добавить `--log-commands` флаг или `logging` выполняемых команд
3. M-02: Передать токенизатор как параметр в splitter вместо глобального state
4. M-04: Заменить `os.chdir` в тестах на `monkeypatch` или параметризацию путей

**Общий уровень:** код зрелый, архитектура устойчивая, основные проблемы — в edge-кейсах параллелизма и auditability.
